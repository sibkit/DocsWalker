# stg-0003 — cli-output-optimization

**Статус:** завершена

## Задача
Оптимизировать формат вывода CLI под потребление LLM: убрать лишнюю обёртку и шум, унифицировать форму связей, поднять плотность сигнала на токен. Все принципиальные развилки уже согласованы с пользователем (см. «Принятые решения»); реализация — без повторного пересогласования.

## Принятые решения

### 1. Контракт ответа: envelope drop (вариант «b» из обсуждения)

- **Успех — exit 0, stdout — JSON-результат команды напрямую.** Без обёртки `{"ok":true,"result":…}`, без альтернативной обёртки `{"state":"ok",…}`. Что вернула команда — то и в stdout.
- **Ошибка — exit ≠ 0, stderr — плоский JSON `{code, message, path?, hint?, describe_type?}`.** Без вложенного `error: {…}`. stdout пустой.
- **Дискриминатор успех/ошибка — exit-code и поток** (stdout vs stderr). Это уже реализовано в `Output.cs` (`Console.Out` для успеха, `Console.Error` для ошибки) и в `Dispatcher.Run` (return 1 на ошибках) — менять только форму JSON, не маршрутизацию потоков.
- **Поле `applied` для write-команд** (true=записано на FS, false=dry-run) становится top-level-полем результирующего объекта, рядом с `id`/`type`/`title` или другими полями результата. Не отдельный конверт.

Принятые осознанно минусы: контракт stdout-JSON становится «асимметричным» — форма зависит от команды, общего wrapper'а нет. Это компенсируется тем, что LLM получает явный контракт через `get-usage-guide`, в котором описана форма результата каждой команды.

### 2. `get-refs` под единый формат `map<name → ids[]>`

Сейчас `get-refs` отдаёт `in[]/out[]` плоским массивом записей `{name, target_id}`. На 16 связях с одним именем имя дублируется 16 раз; форма расходится с `out_refs` в `get-nodes` и `describe-type` (там уже `{ref-name: [ids]}`).

Новая форма результата `get-refs`: объект с двумя полями `in` и `out`, каждое — map имени связи в массив id целей. Пример идентичен полю `out_refs` в `get-nodes` — единый контракт «связь → список целей» по всей API. Параметр `--name=<filter>` оставляет в обеих map'ах только запись с этим именем (или пустые map'ы, если связей с таким именем нет).

Порядок id внутри списка — по возрастанию (детерминированный, для предсказуемых diff'ов).

### 3. Embedded `describe_type` в ошибках — trim до проблемного ref

Сейчас `ErrorEnrichment.TryDescribeType` отдаёт **весь** контракт типа в поле `error.describe_type`. На `missing_required_ref` это означает, что LLM получает все out_refs ради одного, которого не хватает. На типе `section` это ~1300 байт лишнего контекста на каждой ошибке.

Решение: при ошибках, у которых известно конкретное проблемное имя связи (`missing_required_ref`, `invalid_ref_value` и аналоги), `error.describe_type.out_refs` содержит **только эту связь** (одну запись). Шапка типа (`name`, `description`, `text_required`) остаётся, чтобы LLM не теряла контекст. Если LLM нужен полный контракт типа — она зовёт `describe-type --name=<X>` явно.

Для ошибок, не локализованных конкретной связью (валидация параметров CLI и т.п.) — поведение прежнее, полный describe_type.

### 4. Опускание дефолтов и пустот

Не сериализуются:

- `cardinality:"many"` и `required:false` в RefDef (`describe-type`, `get-schema`) — это дефолты схемы. На tree-refs они и сейчас не пишутся; здесь распространяем правило на non-tree refs. При чтении YAML отсутствие поля = дефолт.
- `children: []` в `get-map` и `get-subtree` — узел-лист не несёт пустого массива.
- `subtree_tokens` если он равен `tokens` — на листе всегда равен; LLM при отсутствии поля считает узел листом и подставляет `subtree_tokens = tokens`. Это правило фиксируется в LLM-guide.
- `parameters: []` и `examples: []` в `get-usage-guide` — у команд без параметров/примеров эти поля не выводятся.

## Итоговый порядок выполнения

1. **`cli-output-spec`** — docs первыми (правило проекта `CLAUDE.md`: docs/ — единственный источник правды; код синхронизируется с docs). Шаг не трогает код.
2. **`drop-cli-envelope`** — самая крупная structure-changing правка; делаем раньше остальных, чтобы дальнейшие шаги ложились на новую форму вывода. Сюда же — синхрон `UsageGuideText.MentalModel` под новый контракт.
3. **`get-refs-map-form`** — отдельный contract-changing для одной команды. Изолировано в `ReadApi.GetRefs` и его сериализации.
4. **`omit-defaults-empties`** — точечные cleanup'ы по сериализаторам Schema/ReadApi/UsageGuide. После крупных, чтобы не смешивать с envelope-правкой.
5. **`trim-error-describe-type`** — изолированное изменение в `ErrorEnrichment` плюс минимальное расширение write-handler'ов, чтобы пробрасывать `RefName` из Core-ошибок.
6. **`update-stale-cli-examples`** — docs-only ревизия bare-command examples в секции CLI-интерфейс. Не блокирует и не блокируется code-шагами; ставится в конец как cleanup-хвост, замеченный при step-01.
7. **`define-get-subtree`** — docs-only добавление узла-определения get_subtree в секции «Операции чтения». Тоже cleanup-хвост step-01; делаем после `omit-defaults-empties`, чтобы текст определения сразу учитывал новые правила сериализации (`subtree_tokens` опускается при равенстве `tokens`, `children=[]` опускается).
8. **`define-get-ancestors`** — docs-only добавление узла-определения get_ancestors в секции «Операции чтения». Симметричный пробел с `define-get-subtree`: команда живая (`get_ancestors` в R15 stg-0002), а определения в docs нет. Делаем сразу после п.7 — оба определения берут id из `sequence.txt` последовательно одной правкой.

## Точка возобновления (для новой сессии после сброса контекста)

Стратегия — оптимизация формата вывода CLI под LLM. Все принципиальные решения зафиксированы выше (раздел «Принятые решения») — пользователь подтвердил, новая сессия не должна пересогласовывать.

Старт новой сессии:

1. Прочитать проектный `CLAUDE.md` — общие правила (docs/ как первоисточник, atomic git-вызовы, Glider для C#-навигации, автокоммит/автопуш в конце пачки шагов).
2. Прочитать этот файл (`strategy.md`) полностью — особенно «Принятые решения» и «Итоговый порядок выполнения».
3. Прочитать первый `[*]`-step `step-cli-output-spec.md` и работать строго по нему.
4. Дальше — по итоговому порядку (drop-cli-envelope → get-refs-map-form → omit-defaults-empties → trim-error-describe-type).

Что **не** надо пересматривать:

- Envelope drop, вариант «b» (stdout=result, stderr=плоская ошибка, exit-code дискриминатор) — окончательно. Альтернативы (`state` discriminator, гибрид) рассмотрены и отвергнуты.
- `get-refs` map-form — окончательно. Старая форма `[{name, target_id}]` уходит без shim.
- Trim `describe_type` в ошибках — только проблемный ref, окончательно. Полный contract LLM может запросить отдельным вызовом `describe-type`.
- Список опускаемых дефолтов и пустот — фиксированный (см. п. 4 «Принятые решения»). Без расширения.

Реализация пачки шагов — по правилам planning-skill: первым tool-call в `strategy.md` всем шагам пачки `[*] → [.] (NN)` с присвоением последовательных NN и переименованием step-файлов; затем правки кода; в конце пачки — `[.] → [+]` плюс автокоммит/автопуш по правилам проекта.

Размер пачки: каждый шаг укладывается в ~300K токенов сам по себе. Можно пачковать `omit-defaults-empties` и `trim-error-describe-type` вместе (оба мелкие, изолированные); остальные — лучше по одному.
