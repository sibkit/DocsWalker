# stg-0004 — persistent-server-spec

## Цель

Полностью задокументировать в `docs/DocsWalker.yml` архитектуру persistent-server'а так, чтобы все последующие code-шаги были механической реализацией готового контракта. Шаг docs-only, кода не трогает.

## Файлы

`docs/DocsWalker.yml` — добавить узлы:
- определение режима работы (single long-lived process с lock'ом).
- определение dual-mode TTY/headless.
- определение IPC-протокола (форма кадра, handshake, error-форма).
- определение клиент-режима существующих CLI-команд.
- определение FS-layout: `docs/.docswalker/run.lock`, `docs/.docswalker/run.pid`.
- определение команды `run` в секции CLI-интерфейс.
- определение поведения «сервер не запущен» (плоская ошибка `server_not_running`).
- инвариант «эксклюзивный lock — внешний edit требует kill→edit→restart».
- инвариант «client-mode — единственный путь, in-process fallback убирается».
- инвариант «single-root-per-process».
- инвариант «без auto-spawn клиентом».
- инвариант «версия handshake жёсткая, без миграций».

`docs/.docswalker/sequence.txt` — выдать N последовательных id под новые узлы (точное N — по факту вставки).

## Действия

1. Найти подходящую секцию в `DocsWalker.yml` для архитектурного описания (раздел «Архитектура» или эквивалент). Если такой секции нет — обсудить с пользователем создание новой; не молча.
2. Добавить узлы по списку выше — каждый со своим id из `sequence.txt`, с правильными out_refs (используя текущий контракт после stg-0003: ref-name → array of ids).
3. Описать команду `run` в секции «CLI-интерфейс»: параметры (`--root` обязательный, `--quiet` опциональный — режим без баннера для headless), формат stdout (баннер о захвате lock'а и готовности IPC; в headless — просто блокирующее ожидание; в TTY — REPL-prompt), формат ошибок старта (lock уже взят другим pid → `server_already_running`; не удалось взять lock → `lock_failed` с описанием).
4. Описать клиент-режим: для каждой не-`run` команды путь — найти pid-файл, коннект в IPC, отправка запроса, проксирование ответа. Без сервера — плоская ошибка `server_not_running`.
5. Описать поведение stale `run.pid` (pid в файле, но процесса с таким pid нет / процесс не наш): при старте `run` — перезаписать; при клиент-режиме — трактовать как «сервер не запущен».
6. Зафиксировать форму wire-кадра request/response: newline-delimited JSON, поля запроса (`cmd`, `args[]`), поля ответа (тот же объект, что текущий stdout JSON / тот же объект, что текущий stderr плоский error из stg-0003) + поле-дискриминатор `kind: "ok" | "error"`.
7. Зафиксировать handshake: первое сообщение клиента — `{client_version, protocol_version}`; ответ сервера — `{server_version, accepted: bool, reason?: string}`.
8. Запустить `docswalker check-integrity --root=docs/` после правок (через клиент-режим к запущенному `run` — тут возникает chicken-and-egg, но `check-integrity` пока работает по старой in-process модели до шага 5; то есть через build'еный binary).
9. Прогнать визуальный review секции CLI на консистентность с принятыми решениями (нет ли остатков «запустился — обработал — вышел» в общих описаниях).

## Риски

- Объём узлов значительный: при первом проходе может вылезти много мелких пробелов в существующих описаниях (например, формулировки в «общих принципах» противоречат новой модели). Все такие пробелы фиксируем тут же одной правкой — оставлять «знание-в-чате» нельзя.
- Возможно потребуется сменить формулировку в `UsageGuideText.MentalModel`, если она сейчас говорит о per-call cold-start. Если нужна — правка УЖЕ в этом шаге, чтобы docs и runtime-guide не разошлись.
- Размер `DocsWalker.yml` после правки растёт. Это нормально, но проверить, что AOT-парсер не упирается в лимиты (вряд ли, но проверить).
