# stg-0011 — migrate-classifiers-data

## Цель

Проставить классификаторы на все существующие 337 узлов `docs/`, чтобы граф снова стал валидным после введения required tree-связей классификаторов. Стратегия: массово `none` для всех, точечно — реальная классификация по семантически очевидным случаям (subject/subsystem можно вывести из секции, в которой узел лежит; audience — из типа узла или содержимого; `csharp_structure` — только для узлов про конкретный код, остальное `none`).

## Файлы

`docs/` через MCP — bulk `transaction`.

## Действия

1. После шага `docs-schema-classifier-trees` граф невалиден. Зафиксировать факт через `check-integrity` (ожидаемые массовые `missing_classifier`).
2. Массовая фаза: `transaction` с `create-ref` от каждого атом-узла к `none` в каждом классификаторе. Большой пакет операций, возможно — несколько `transaction`-вызовов по сериям.
3. Точечная фаза: проход по разделам `docs/`, для очевидных случаев менять `none` → конкретная категория через `redirect-refs` или `delete-ref`+`create-ref`.
   - Узлы из section `Операции чтения` (id=17) → `subject=api/read`.
   - Узлы из section `Операции записи` (id=27) → `subject=api/write`.
   - Узлы из section `MCP-интерфейс` (id=44) → `subsystem=mcp`.
   - Узлы из section `CLI-интерфейс` (id=35) → `subsystem=cli`.
   - Узлы `llm_hint` → `audience=llm-agent`.
   - И аналогично.
4. После каждой фазы — `check-integrity`. После всех — `errors=[]`.

## Риски

Очень большой объём изменений. Делать через `--dry-run=true` сначала, валидировать форму ответа, потом без `dry-run`. При ошибке `transaction` — атомарный rollback гарантируется самим transaction-движком. Точечная фаза может затянуться — допустимо разнести в несколько пачек реализации.
