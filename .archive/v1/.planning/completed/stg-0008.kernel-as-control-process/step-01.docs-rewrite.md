# stg-0008 — step-01 — docs-rewrite

## Цель

Переписать узлы `docs/DocsWalker.yml` под архитектуру **kernel-as-control-process**. Существующий граф описывает старую модель (per-root server, IPC через named pipe, sole-владение `docs/`, lock на данные, single-root-per-process). Новый граф должен описывать:

- один kernel на пользователя (multi-root внутри);
- HTTP + JSON-RPC 2.0 транспорт;
- per-user discovery (`%LOCALAPPDATA%\DocsWalker\kernel.json` / `$XDG_RUNTIME_DIR/docswalker/kernel.json`);
- sole-writer без file-lock на данные (отказ от #359);
- per-root idle eviction (10 мин default);
- kernel always-running (без kernel-level idle);
- MCP-wrapper как тонкий stdio↔HTTP bridge с фиксированным `--root=`;
- auto-spawn разрешён, но **не silent**;
- `run` → `repl`, HTTP-клиент;
- per-root semaphore (вместо global #313).

Шаг **не трогает код** — все правки только в `docs/DocsWalker.yml` через DocsWalker `transaction`.

## Файлы

- `docs/DocsWalker.yml` — все нодные правки.
- `docs/.docswalker/meta-schema.yml` — **не трогаем** (мета-схема описания узлов остаётся как есть).

## Действия

1. **Sanity-check CLI** — убедиться, что бинарь `DocsWalker.Cli.exe` (publish) собран и отвечает (например, `get-usage-guide`). Если бинаря нет — собрать (`dotnet publish`).

2. **MCP диагностика (отдельной заметкой)** — в текущей Claude-сессии docswalker MCP не виден среди инструментов. Зафиксировать это в чате как обратную связь о DocsWalker (CLAUDE.md: «обратная связь о DocsWalker»). Не блокирует step-01 — работаем через CLI.

3. **Inventory старых узлов** — через CLI вытянуть содержимое узлов из списка зацепленных стратегией: #304, #305, #307, #308, #309, #310, #311, #312, #313, #314, #316, #317–#321, #322–#324, #325–#332, #333, #359, #367, #372. Использовать `get-nodes` с пакетом id, или `get-subtree` по корневому узлу «Модель процесса».

4. **Карта правок** — для каждого узла определить: `update` (переформулировать содержимое), `delete` (выбросить как класс — например, лок-семантика #308, IPC named-pipe #310, single-root #309, checksum #359), либо `leave` (если узел не противоречит новой модели). Список новых узлов с парентом, ссылкой на корень «Модель процесса» (или нового подкорня) и телеграфным описанием.

5. **Проверка обратных ссылок** — для удаляемых узлов через `get-in-refs` найти, кто на них ссылается. Эти ссылки — редиректнуть (`redirect-refs`) или удалить (`delete-ref`).

6. **Транзакция** — собрать атомарную `transaction` (update_node / create_node / delete_nodes / redirect_refs). Применить.

7. **Пост-проверки:**
   - `check-integrity` — граф консистентен.
   - `search` по ключевым словам старой модели («named pipe», «server_not_running», «single-root», «exclusive lock на docs», «checksum invalidation») — убедиться, что старых упоминаний не осталось (или явно объяснены как «исторический контекст»).
   - `get-subtree` на корне «Модель процесса» — визуально проверить, что новая модель читается логично.

8. **Сверка со strategy.md** — пробежать «Принятые решения» из `strategy.md` и убедиться, что каждое decision отражено в соответствующем узле docs.

## Риски

- **Объём правок большой** — много узлов, легко пропустить ссылку. Mitigation: атомарная транзакция (либо всё применилось, либо ничего), плюс post-проверки `check-integrity` + `search` по ключевым словам.
- **Удаление узлов с входящими ссылками** — если на удаляемый #309 (single-root) ссылается LLM-guide или example, нужно либо переформулировать ссылающийся узел, либо `redirect-refs` на новый узел («multi-root в одном ядре»).
- **Sequence/ord** новых узлов — чтобы они встали в логический раздел документа (рядом с «Модель процесса», не в конце файла). Проверить через `get-subtree` после.
- **DocsWalker MCP не виден в сессии** — диагностика причины (бинарь не собран? lock-конфликт? crash?) выходит за рамки step-01, фиксируется как обратная связь. Step-01 идёт через CLI без блокировки.
