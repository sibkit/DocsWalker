# stg-0006 — root-as-entry-point

**Статус:** завершена

## Задача

`Node.RootId = 0` сегодня — sentinel: id=0 валидно как target в `out_refs.path` каждого top-level документа, но `get-nodes ids=[0]` отдаёт `node_not_found` (`ReadApi.cs:392` — «root присутствует в графе только концептуально»). Для LLM это «ссылка на узел, которого нет», подрывает доверие к out_refs. Сделать root полноценным узлом графа на чтение: тип `root` в Схеме, синтезированный Node через `Graph.GetById(0)`, явные write-блокировки. Точка входа для LLM: наткнувшись на `path:[0]` в любом ответе, она может зафетчить id=0 и получить pointer на `get-usage-guide`.

## Принятые решения

### 1. text — минимум

`root.text` — одна строка-pointer на `get-usage-guide`. Без mental-model, без перечня top-level документов. Mental-model уже несёт `get-usage-guide`, дублирование вредно (токены, расхождение источников правды).

### 2. title — `"docs"`

Соответствует FS-имени папки docs/. Альтернатива `"DocsWalker"` (имя проекта) отвергнута — `title` это path-сегмент, а path начинается с пустой строки до первого документа; «docs» делает это явным.

### 3. `get-map` — без изменений

Сейчас `get-map` возвращает плоский список документов. Перевести его на старт от id=0 (root в шапке, документы как children) — отдельная задача. На этом шаге не меняем: root-as-entry и get-map — разные access patterns, пусть сосуществуют.

### 4. `out_refs` у root — пустой

В Схеме у `root` объявляется `out_refs: []`. Top-level узлы (документы + folders) находятся через стандартный механизм: `get-in-refs --id=0 --name=path`. Это работает «из коробки», т.к. обратный индекс строится из `out_refs.path:[0]` всех документов; после того как `GetById(0)` перестанет возвращать null, in-refs запрос станет валидным без отдельной поддержки.

### 5. Синтез на лету, без персистенции

Root **не пишется** в YAML, **не хранится** в `_byId`. `Graph.GetById(0)` каждый раз возвращает свежий Node-объект, собранный из констант (id, type, title, text — литералы). Документы и folders по-прежнему ссылаются на id=0 через `out_refs.path`; обратный индекс уже есть. Write-API остаётся untouched — root как ресурс не изменяем.

### 6. Write-блокировки — через явный код ошибки `cannot_modify_root`

Любая попытка `update-node id=0`, `delete-nodes ids=[..,0,..]`, `move-node id=0`, `create-ref from-id=0` или `delete-ref from-id=0` отвергается с `cannot_modify_root` и подсказкой «root — синглтон, изменению не подлежит». `delete-nodes id=0` уже сейчас возвращает ошибку (`WriteApi.cs:858`), но текст у неё «Корневой узел id=0 удалить нельзя» — приведём к единому коду `cannot_modify_root`.

### 7. Бамп мета-схемы v5 → v6

Текущая мета-схема `docs/.docswalker/meta-schema.yml` (constraint строки 35) запрещает объявлять `root` как тип; одновременно constraint строки 82 ссылается на «тип root» как на возможный кейс. Несамосогласованность — реликт sentinel-модели. В рамках этой стратегии:

- `meta_schema_version: 5 → 6`.
- Из constraint строки 35 убирается «Не объявляется как тип» — остаётся «Имя `root` зарезервировано за корневым синглтоном (узел id=0)».
- Constraint строки 82 («`name=path` обязана у каждого type_definition кроме типа root») сохраняется как есть — теперь с реальным смыслом.
- Парсер мета-схемы и валидатор Схемы перестают спецкейсить «root undeclared».

Альтернативы (синтез без декларации в Схеме / гибрид с спецкейсом в `describe-type`) рассмотрены и отвергнуты: первая оставляет UX-яму на `describe-type --name=root`, вторая создаёт два источника правды о типах. Bump до v6 — однократная цена за чистоту.

   - Парсер мета-схемы (`MetaSchemaLoader`) поднимает поддерживаемую версию до 6 и снимает спецкейс «root undeclared».
   - `docs/DocsWalker.yml`: 2–3 узла с семантикой id=0 как точки входа (через `docswalker create-node`).
   - Acceptance: `check-integrity` ok, `describe-type --name=root` отдаёт контракт нового типа.

2. **root-synthesis** — код поведения:
   - `Graph.GetById(0)` синтезирует Node `{0, "root", "docs", "<pointer на get-usage-guide>", out_refs={}}`.
   - Существующие гарды по `Node.RootId` в `WriteApi` приводятся к коду `cannot_modify_root` единым сообщением.
   - `UsageGuideText.cs` дополняется одной строкой про id=0 как точку входа.
   - Тесты: `GetNodes ids=[0]` отдаёт root; `GetInRefs id=0 name=path` возвращает все top-level документы и folders; check-integrity ok; existing-тесты, которые ожидали `GetById(0) == null`, обновляются.

Каждый шаг — отдельная пачка. После [+] каждой пачки — автокоммит/автопуш по правилу проекта.
