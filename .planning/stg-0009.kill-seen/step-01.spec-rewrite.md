# stg-0009 — step-01 — spec-rewrite

## Цель

Переписать `docs/DocsWalker.yml` под модель «никакого server-side seen». Все узлы про context-awareness (placeholder, seen-set, session-state на диске, TTL, write-invalidation, reset на guide), про источники session_id и про MCP-handshake-передачу session_id — удалить. В описаниях read-команд явно зафиксировать новый контракт: повторы всегда полные, для дешёвого обзора — `fields=[title]` + `depth`/`tree`. Узлы про auto-include (#341, #363) — **остаются**, это другой механизм.

Шаг **не трогает код** — все правки только в `docs/DocsWalker.yml` через DocsWalker `transaction`.

## Файлы

- `docs/DocsWalker.yml` — все правки.
- `docs/.docswalker/meta-schema.yml` — **не трогаем** (мета-схема описания узлов).

## Действия

1. **Sanity-check CLI и kernel** — последовательность:
   - Проверить, не запущен ли уже kernel (старая dev-сборка может висеть): `KernelInfoFile.TryRead()` лежит в per-user папке (см. `KernelDiscovery.GetKernelInfoPath`); если есть и pid жив — это **другая** сборка, и она ответит первой. Корректно: остановить старый процесс (taskkill на pid) **или** дождаться idle-eviction (10 мин на root). Если старый — это наш-же published-bin прошлой сессии и его поведение уже корректное, можно и не трогать.
   - `dotnet publish` обоих exe, если ещё не собраны.
   - Запустить `docswalker get-usage-guide --root=<repo>` через published-бинари — убедиться, что CLI поднимает kernel и отвечает.
   - Прочитать форму payload транзакции: `docswalker describe-type transaction --root=<repo>` (или соответствующий `get-meta-schema` запрос). Без этого собирать `transaction` вслепую — риск отказа на этапе применения.
   - **Fallback**: если транзакция через CLI не работает по техническим причинам — Edit на YAML с явной отметкой в чате что это фоллбэк (нарушение приоритета DocsWalker, но единственный способ продолжить).

2. **Inventory** — через CLI:
   - `get-nodes --root=<repo> --ids=342,343,344,345,346,347,348,349,350,351,352,353,354,355,358,359,360,361,362,370,371,373,374` (батчем через запятую) — собрать содержимое всех удаляемых узлов.
   - `get-by-path --root=<repo> --path=<path-к-context-awareness>` — посмотреть структуру родительского раздела.
   - `search --root=<repo> --query=session_id` — найти все узлы (description'ы команд, рамочные документы), где встречается `session_id` или `seen` (последнее по запросу `seen-set`, `--no-seen`, `placeholder`).

3. **Inbound-refs analysis** — для каждого удаляемого узла:
   - `get-in-refs --root=<repo> --id=<N>` — найти, кто ссылается. 
   - Решить по каждому inbound: 
     - Если ссылающийся узел — example у удаляемого rule, удалить его вместе.
     - Если ссылающийся узел — другой rule, который вообще-то остаётся, удалить только конкретную связь (`delete-ref`) или переформулировать (`update-node`).
     - **Важно:** проверить, не ссылается ли auto-include (#363) или пример #341 на удаляемые — не должны, но проверить.

4. **Карта правок** — оформить отдельно (в чате или scratch-pad), на каждый узел: `delete` / `update` / `redirect-refs` / `delete-ref`. На каждый новый/правленый текст — короткая заметка о новой формулировке.

5. **Транзакция** — собрать атомарную `transaction` (массив операций: `delete_nodes`, `update_node`, `delete_ref`, `redirect_refs`). Применить через `transaction --root=<repo>`. Атомарность гарантирует: либо все правки применились, либо ничего.

6. **Пост-проверки:**
   - `check-integrity --root=<repo>` — граф консистентен (нет битых рефов после удалений).
   - `search --root=<repo> --query=session_id` — пусто (или только остатки в исторических контекстах, которые мы решили оставить).
   - `search --root=<repo> --query=seen-set` — пусто.
   - `search --root=<repo> --query=placeholder` — только если placeholder упоминается в другом контексте (например, JSON-маршалинг). Проверить вручную.
   - `search --root=<repo> --query=--no-seen` — пусто.
   - `get-nodes --root=<repo> --ids=363,341` — auto-include-узлы на месте.
   - `get-nodes --root=<repo> --ids=338` — заголовок «Auto-include», #339 переписан, #340 без упоминаний seen/placeholder.
   - **Функциональный auto-include**: `get-subtree --root=<repo> --id=125` (rule «Запись через API») — в payload есть `auto_includes` с узлом #188. Проверка, что #363 как механизм работает после удаления seen-обвязки.
   - **Замечание про get-usage-guide**: post-check `get-usage-guide --root=<repo>` перенесён в step-04 (smoke). На текущем шаге текст guide хардкоден в `UsageGuideText.cs` и со старыми формулировками — это нормально, правки кода — следующий шаг.

7. **Сверка со strategy.md** — пробежать раздел «Принятые решения» (особенно #1 session_id целиком, #4 что остаётся) и убедиться, что:
   - Узлы про session_id удалены (решение #1 отражено).
   - `--no-seen` не упоминается в описаниях команд (#2 отражено в spec; код — следующий шаг).
   - Auto-include и MCP-server упомянуты как **сохраняемые** (#4 отражено).
   - Замена в usage-guide (#5) — присутствует.

## Риски

- **Большой объём узлов** — 13 rules + 3 definitions + N родителей + правки в описаниях команд. Mitigation: атомарная `transaction` + post-проверки `search` на отсутствие старых ключевых слов.
- **Отказ `transaction` на применении** — например, неучтённый inbound-ref, integrity-violation, синтаксическая ошибка в payload. Атомарность гарантирует rollback (граф нетронут). Mitigation: разобрать сообщение об ошибке, дополнить карту правок (новый `delete-ref`/`update-node`), повторить. **Не** обходить через `Edit` YAML — это сломает другую цель (dogfooding).
- **Inbound refs на удаляемые** — наверняка есть (например, описание команды `get-nodes` ссылается на `#350 --no-seen`). Mitigation: пройти по каждому inbound через `get-in-refs` перед применением транзакции.
- **CLI/kernel недоступен** — Debug-бинари путей могут не находить друг друга (видели на смоке stg-0008). Mitigation: `dotnet publish` обоих exe, использование published-бинарей; крайний фоллбэк — Edit YAML с отметкой.
- **Родительский раздел остаётся пустым** — если у «Context-awareness» все дочерние удалены, родитель — пустой контейнер. Решение: либо удалить родителя, либо переписать его telegram-style («Сервер не ведёт state между запросами; для дешёвого обзора — `fields`/`depth`/`tree` в read-командах»). Решение по месту.
- **MCP-сервер не виден в Claude-сессии** — как и в step-01 stg-0008, MCP-канал DocsWalker может быть не зарегистрирован. Не блокирует — работаем через CLI. Зафиксировать в чате как обратную связь, не как ошибку.
