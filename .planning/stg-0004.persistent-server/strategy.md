# stg-0004 — persistent-server

**Статус:** текущая

## Задача

Ввести модель long-lived процесса DocsWalker, который при старте захватывает эксклюзивный lock на `docs/`, парсит YAML и держит граф в RAM до выхода. Все существующие CLI-команды переходят в клиент-режим: парсят argv, отправляют запрос локальному серверу через named pipe (Windows) или Unix socket (POSIX), проксируют ответ. Cold-start per-call уходит. Дополнительно тот же серверный процесс при запуске в TTY даёт REPL-prompt — единый бинарь, два интерфейса (TTY → REPL, headless → daemon, IPC всегда).

## Принятые решения

### 1. Хранилище — папка YAML, без архива

`docs/` остаётся редактируемой папкой YAML. Это сохраняет: прямой человеко-edit в IDE, git-blame и git-diff построчно, согласованность с правилом проектного `CLAUDE.md` про docs/ как источник правды. Бинарный архив рассмотрен и отвергнут — ломает всё перечисленное ради сомнительной выгоды.

### 2. Эксклюзивный lock на всё время жизни процесса — feature, не bug

Сервер захватывает file-lock на `docs/` (через файл `docs/.docswalker/run.lock`) при старте, держит до выхода. Внешние правки YAML руками на работающем сервере **не разрешены** по контракту — пользователь сам гасит сервер, правит, поднимает обратно. Это упраздняет FS-watcher, исключает race-условия между «правкой через CLI» и «правкой в IDE», даёт чёткое единоличное владение хранилищем.

### 3. Dual-mode: TTY → REPL, headless → daemon

Один процесс, два интерфейса для владельца терминала: при `Console.IsInputRedirected == false` — readline-prompt с теми же argv-командами, что и one-shot CLI; при редиректе stdin / запуске без TTY — блокирующее ожидание сигнала на завершение. Внешний IPC-канал работает в обоих режимах одинаково.

### 4. IPC — только локальный, named pipe / Unix socket

Windows — `\\.\pipe\docswalker-{root-hash}`. POSIX — `${XDG_RUNTIME_DIR}/docswalker-{root-hash}.sock` (fallback `/tmp/docswalker-{root-hash}.sock`). ACL «только владелец процесса». TCP не открываем — снимает целый класс сетевых угроз (auth, encryption, экспозиция в LAN). Хеш `root`'а в имени канала — namespacing нескольких workspace'ов на одном хосте.

### 5. Client-mode — единственный путь существующих команд

Все команды кроме `run` уходят в клиент-режим: бинарь стартует → читает `docs/.docswalker/run.pid` → коннектится в IPC → шлёт запрос → проксирует ответ → выходит. Если `run.pid` нет или сервер не отвечает — плоская ошибка `server_not_running` + exit ≠ 0 + хинт «запусти `docswalker run --root=…`». Старый in-process путь убирается **полностью**: два режима работы (через сервер / напрямую) удвоили бы стоимость сопровождения. CI и one-shot use-case'ы делают явный pipeline `docswalker run &` → запросы → kill.

### 6. Без auto-spawn'а сервера клиентом

Клиент не запускает сервер за пользователя. Auto-spawn создаёт фоновые процессы «откуда-то» — непрозрачно, не понятно кто их убирает за собой, непредсказуемо в CI. Лучше внятная ошибка с готовой командой запуска.

### 7. Single-root-per-process

Один сервер обслуживает один root. Несколько workspace'ов на хосте = несколько серверов, у каждого свой `run.pid`, свой pipe/socket, свой namespace. Multi-root внутри одного процесса рассмотрен и отвергнут — добавляет диспетчеризацию по root'у в каждый запрос ради сомнительной экономии процессов.

### 8. Запросы сериализуются на сервере

Сервер обрабатывает запросы строго по одному (per-channel и глобально). Граф read-mostly; параллельные read'ы потребовали бы `RWLock` и аккуратной синхронизации mutable-state при write-командах. Если когда-нибудь упрёмся в производительность — введём read-параллелизм без смены wire-протокола (это деталь реализации сервера, не контракта).

### 9. REPL — без persistent history между сессиями

В рамках одной сессии — нативный line-edit, что доступно из .NET 10 на консоли. Между сессиями история не сохраняется — для MVP не нужно, добавится отдельной задачей при потребности.

### 10. Жёсткий version handshake без миграций

Первое сообщение клиента после коннекта — `{client_version, protocol_version}`. Сервер сверяет; mismatch → ошибка «client/server version mismatch, restart server» + exit ≠ 0. Никаких compat-shim'ов. Бинарь клиента и сервера деплоятся вместе одним артефактом — рассинхрон возможен только если кто-то держит старого клиента, явно ошибка пользователя.

## Итоговый порядок выполнения

1. **persistent-server-spec** — docs первыми (правило проекта `CLAUDE.md`: docs/ как первоисточник, код синхронизируется с docs). Полная архитектурная фиксация модели в `docs/DocsWalker.yml`. Code-шаги ложатся на готовый контракт. Шаг не трогает код.

2. **server-lifecycle** — фундамент серверного процесса: lock-файл, pid-файл (со stale-detection), низкоуровневый IPC-handle (named pipe Win / Unix socket POSIX), graceful shutdown по SIGINT/SIGTERM/Ctrl+C. Без бизнес-логики команд — только каркас «процесс взял ресурсы, держит, корректно отпускает». Все следующие code-шаги стоят на этом каркасе.

3. **ipc-protocol** — request/response framing поверх handle'а из шага 2: newline-delimited JSON-frames, version handshake, серверная маршрутизация принятого запроса в существующий `Dispatcher.Run`, клиентская функция отправки. Без команды `run` и без переключения существующих CLI — только серверная и клиентская плёнки байтов.

4. **run-command-and-tty-repl** — собственно команда `docswalker run`: захват lifecycle (шаг 2) + старт IPC-сервера (шаг 3) + ветвление по `Console.IsInputRedirected`. TTY → REPL-loop (readline → парс argv → тот же Dispatcher → печать ответа). Headless → блокирующее ожидание сигнала. Регистрация команды в `Commands.cs`, описание в `UsageGuideText`.

5. **client-mode-default** — переключение всех остальных CLI-команд: на входе бинарь смотрит cmd; если cmd=`run` — серверный путь (шаг 4); иначе — клиент-режим через IPC. Старый in-process Dispatcher-путь для не-`run` команд удаляется полностью. Финал — новая модель становится единственной.

Каждый шаг — отдельная пачка. Шаги связные (каждый стоит на предыдущем), параллелизация невозможна. После `[+]` каждой пачки — автокоммит/автопуш по правилу проектного `CLAUDE.md`.

## Точка возобновления (для новой сессии после сброса контекста)

Стратегия — вводит long-lived процесс DocsWalker с эксклюзивным владением `docs/` и переводит все CLI-команды на клиент-режим через локальный IPC. Все принципиальные решения зафиксированы выше (раздел «Принятые решения») — пользователь подтвердил, новая сессия не пересогласовывает.

Старт новой сессии:

1. Прочитать проектный `CLAUDE.md` — общие правила (docs/ как первоисточник, atomic git-вызовы, Glider для C#-навигации, автокоммит/автопуш в конце пачки шагов).
2. Прочитать этот файл (`strategy.md`) полностью — особенно «Принятые решения» и «Итоговый порядок выполнения».
3. Прочитать первый `[*]`/`[.]`-step и работать строго по нему.
4. Дальше — по итоговому порядку, шаг за шагом, без пропусков.

Что **не** надо пересматривать:

- Хранилище = папка YAML, не архив. Окончательно.
- Эксклюзивный lock на всё время жизни сервера = feature. Внешний edit YAML руками требует kill→edit→restart. Окончательно.
- Dual-mode (TTY → REPL, headless → daemon) — единый процесс с TTY-detection. Окончательно.
- IPC только локальный (named pipe / Unix socket). TCP не открываем. Окончательно.
- Client-mode — единственный путь существующих команд. In-process fallback убирается полностью. CI делает `run &` → запросы → kill. Окончательно.
- No auto-spawn клиентом. Окончательно.
- Single-root-per-process. Окончательно.
- Сериализованная обработка запросов. Окончательно.
- REPL без persistent history между сессиями. Окончательно.
- Жёсткий version handshake без compat-shim'ов. Окончательно.

Реализация пачки шагов — по правилам planning-skill: первым tool-call в `strategy.md` всем шагам пачки `[*] → [.] (NN)` с присвоением последовательных NN и переименованием step-файлов; затем правки кода; в конце пачки — `[.] → [+]` плюс автокоммит/автопуш.

Размер пачек: каждый шаг укладывается в ~300K токенов сам по себе. Пачковать шаги между собой нельзя — каждый стоит на предыдущем; идём по одному.
