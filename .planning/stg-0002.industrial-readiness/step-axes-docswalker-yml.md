# stg-0002 — axes-docswalker-yml

## Цель
Переписать `docs/DocsWalker.yml` под концепт «ось». Снять понятия «системный ref_type», «связь по умолчанию», «прикладной ref_type» — заменить единым «ось» с тремя `kind` (`system`/`default`/`explicit`). Удалить `list_documents` и упоминания `create_document`/`delete_document` — их функцию закрывают `create_node`/`delete_node` через ось `path`.

## Файлы
`docs/DocsWalker.yml` — корневая правка по разделам:

### Раздел (#7) «Модель данных»
- Удалить definitions `(#10) тип связи (ref_type)`, `(#11) системный тип связи`, `(#12) прикладной тип связи`, `(#13) связь по умолчанию (default ref)`.
- Ввести вместо них:
  - `(#10) ось (axis)` — направленное именованное отношение, объявленное в Схеме.
  - `(#11) kind оси: system|default|explicit`.
  - `(#12) cardinality оси: one|many`.
  - `(#13) обязательная ось типа` — ось из `must_have_axes` типа узла; её значение требуется при создании узла.
- В rules переписать пункты:
  - «Все типы прикладных связей объявлены как ref_type» → «все оси объявлены в секции `axes` Схемы».
  - «Связи path и связи по умолчанию являются производными» → «оси kind=system и kind=default не хранятся в YAML, выводятся ядром (system — из иерархии path, default — из принадлежности узла child-block родителя)».
  - origin в результатах get_refs — переименовать в `axis_kind` (значения: system | default | explicit).

### Раздел (#16) «Контракт валидации»
- Объединить проверки `refs` и `parent_block` в одну: «для каждого node-типа все обязательные оси (`must_have_axes`) имеют валидное значение по контракту оси (target_types, cardinality)».
- Снять отдельный код ошибки `parent_block_inconsistent` — его покрывает `axis_value_invalid`.
- Сохранить `unique`, `sequence`, `style` без изменений.

### Раздел (#17) «Операции чтения»
- Удалить `(#18) list_documents` — её роль закрывает `get_map` (даёт оглавление по оси `path`) или `get_nodes` с фильтром по type.
- В `(#22) get_nodes` уточнить: поле `out_refs` теперь возвращает значения по всем осям (с `axis_name` и `axis_kind`); поле `parent_id` остаётся как удобный shortcut на значение оси `path`.
- В `(#24) get_refs` / `(#25) get_in_refs` — `type` параметр переименовать в `axis` (для совместимости можно оставить `type` как алиас).

### Раздел (#27) «Операции записи»
- `(#28) create_node` — переписать: параметры — `type`, `title`, плюс **значения обязательных осей по типу из Схемы** (для типа `section` это `path`; для типа `definition` — `path` с обязательным `block=definitions`). Удалить упоминание `parent_id` как фиксированного имени.
- Удалить `(#33) add_ref_type` — заменяется на `add_axis` (новая операция: добавить explicit-ось в Схему).
- `(#31) create_ref` — параметр `type` переименовать в `axis`.
- `(#32) delete_ref` — параметр `type` переименовать в `axis`.
- В `transaction` (#34) — обновить список допустимых операций (убрать add_ref_type, добавить add_axis).

### Раздел (#35) «CLI-интерфейс»
- Заменить пример `(#41)` на `docswalker create-node --type=section --path="DocsWalker/CLI" --title="..."`.
- Добавить пример создания папки и документа: `docswalker create-node --type=folder --path="" --title="Архив"`, `docswalker create-node --type=document --path="Архив" --title="Отчёт" --description="..."`.
- Заменить `(#42) add-ref-type` на `add-axis` с тем же духом.

## Действия
1. Переписать раздел «Модель данных» под ось.
2. Сократить «Контракт валидации» — объединить refs/parent_block.
3. Удалить `list_documents`.
4. Переписать `create_node` под обязательные оси.
5. Переименовать `add_ref_type` → `add_axis`; `create_ref/delete_ref` параметр `type` → `axis`.
6. Обновить CLI-примеры.
7. Сверить с CLAUDE.md проекта: docs/ — единственный источник истины, никаких упоминаний документов в API.

## Риски
Раздел `(#7)` сильно перепишется — нужно сохранить нумерацию id уже выданных пунктов (#10..#13 переиспользуются под новые determinations). Если строгая нумерация не требуется — выдать новые id через sequence.
