# stg-0002 — docs-llm-guide

## Цель
Раздел «Как LLM работает с DocsWalker» в `docs/DocsWalker.yml`: ментальная модель (граф + tree-scopes), порядок работы (read → describe → plan → write), правила удаления, правила добавления связей, типичные ошибки. Доменная документация — отдельно от runtime-инструкции `usage-guide`.

## Структура раздела

1. **Модель**
   - Граф = узлы + named-связи (out_refs).
   - Некоторые связи объединены в named-tree (scope), их объявление — в schema (`tree: <scope>`).
   - Дерево формируется одной или несколькими связями с одинаковым scope-name (могут быть на разных типах с разными именами связей).
   - `path` — встроенный scope, единственный физически материализованный (в файлах/каталогах). Все остальные деревья — навигационные view'ы поверх графа.
   - LLM **не работает с файлами и не знает о них** — только узлы и связи.

2. **Точки входа в граф**
   - Корень — узел с id=0, тип `root`.
   - Вся структура достаётся от root через `get_subtree(0, tree="path")` или последовательно через `get_refs(0)`.
   - Команды `list-documents` нет — документы получаются обходом path-дерева с фильтром по типу.

3. **Порядок работы перед записью**
   - Прочитать актуальное состояние интересующего поддерева (`get_subtree`).
   - При сомнениях по контракту типа — `describe-type`.
   - Спланировать transaction (если изменений много).
   - При первом подходе к незнакомой задаче — выполнить с `--dry-run` и проверить ответ перед реальной записью.
   - Перед стартом сессии — `check-integrity`.

4. **Правила удаления**
   - Только `delete-nodes --ids=<csv>` — список id явный.
   - Авто-каскада нет: LLM сама собирает набор через `get_subtree` (по любому scope) + добавляет path-children каждого узла набора.
   - Если есть входящие cross-refs от узлов снаружи набора — сначала `redirect-refs` или `update-ref` в transaction, потом удаление.
   - Ошибка `path_orphans_left` — забыты path-children, добавь их в набор.
   - Ошибка `dangling_refs` — есть внешние cross-refs, переподшей или разорви их.

5. **Правила связей**
   - Новый out_ref у существующего узла — `create-ref` (с указанием name из контракта типа).
   - Перенаправить группу входящих cross-refs — `redirect-refs` (одной командой).
   - Tree-refs (`path`, `strategic`, …) **не редактируются через redirect-refs** — для них есть `move-node --tree=<scope>`.
   - Объявить новую связь в Схеме — это правка `docs/Схема.yml` человеком, **не задача LLM**. У LLM нет команды «добавить тип ref'а в Схему»: это структурная эволюция, а не runtime-операция.

6. **Запреты**
   - Прямая правка YAML в обход API запрещена (sequence.txt, целостность графа, atomic-write).
   - Cascade-удаление по доменному дереву (например, strategic) — только через explicit `delete-nodes` с собранным набором, без команды-shortcut.
   - Делать `move-node` без указания `--tree`, если намерение — переподшить узел в доменном дереве (default `path` запустит реструктуризацию хранилища).

## Файлы
- `docs/DocsWalker.yml` — новый раздел «Как LLM работает с DocsWalker» (после раздела «MCP-интерфейс»). Структура — 6 пунктов выше, каждый — отдельный statement/note/llm_hint, согласно атомизации R5.

## Действия
1. Спроектировать раздел в виде атомарных узлов (statement/rule/llm_hint).
2. Записать в `docs/DocsWalker.yml`.
3. Прокинуть ссылку на этот раздел из `usage-guide` (когда дойдём до него).
4. Прогнать `check-integrity` — должен пройти.

## Риски
- Только правка docs. Никакого кода. Если правило сформулировано так, что упоминает FS/файлы — переписать (LLM не должна о файлах знать).
- Расхождение с фактическими CLI-командами (имена параметров) — провериться по `Commands.cs` после реализации R12-R16, delete-nodes, redirect-refs.
