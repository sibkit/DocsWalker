# stg-0002 — usage-guide

## Цель
Read-команда `get-usage-guide`: единый manifest, который LLM-агент дёргает в начале сессии. Содержит список команд (имя, параметры, краткая семантика, типичные ошибки), инварианты системы, рекомендуемые паттерны и слепок текущего состояния `docs/` (имена документов, общее число узлов).

## Файлы
`docs/DocsWalker.yml` — раздел «Операции чтения» (#17): новый пункт `get_usage_guide` с описанием формы ответа.
`src/DocsWalker.Cli/UsageGuide/UsageGuideText.cs` (новый) — статическая константа с инструкцией для LLM (русский текст: правила поведения, do/don't, типичные паттерны, ссылки на команды).
`src/DocsWalker.Core/Api/ReadApi.cs` — метод `GetUsageGuide`, собирает manifest из `Commands.All` (через инверсию зависимости — описания команд должны быть доступны из ядра либо передаваться извне) и слепок графа.
`src/DocsWalker.Core/Api/ReadApiJson.cs` — сериализация usage-guide.
`src/DocsWalker.Cli/Cli/Commands.cs` — регистрация команды.
`src/DocsWalker.Cli/Cli/Handlers/ReadHandlers.cs`, `Program.cs`.
`tests/DocsWalker.Tests/ReadApiTests.cs` — тесты.

## Действия
1. Зафиксировать команду в `docs/DocsWalker.yml`.
2. Завести статический текст инструкции (`UsageGuideText`): структурированный объект с разделами «инварианты», «как читать», «как писать», «типичные ошибки и подсказки».
3. Реализовать сборку manifest'а команд из существующих определений (`Commands.All`); описание каждой команды — короткое предложение + список параметров с типами + 1-2 примера.
4. Добавить слепок графа: имена документов и общее число узлов (через готовые методы `ListDocuments`/число записей в `Graph.ById`).
5. Подключить в CLI.
6. Тесты: ответ непуст, содержит ожидаемые секции и команды.

## Риски
Чтобы статика инструкции жила в коде, но описания команд — в `Commands.cs` (CLI), нужен аккуратный API: либо `Commands` переезжает в Core, либо CLI собирает manifest и передаёт в Core, либо Core принимает callback. Решить в начале реализации.
