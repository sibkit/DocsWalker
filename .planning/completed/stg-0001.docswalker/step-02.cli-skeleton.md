# stg-0001 — cli-skeleton

## Цель
Сделать общую обвязку CLI: разбор подкоманд, единая форма JSON-ответа, структурированные ошибки, exit-коды. Логику команд не реализовывать — все команды возвращают «not implemented».

## Файлы
`src/DocsWalker.Cli/Program.cs` — точка входа, диспетчер
`src/DocsWalker.Cli/Cli/ArgParser.cs` — разбор argv, преобразование snake_case ↔ kebab-case, типизированные параметры
`src/DocsWalker.Cli/Cli/Output.cs` — форматирование успешных и ошибочных JSON-ответов в stdout/stderr
`src/DocsWalker.Cli/Cli/Commands.cs` — таблица всех подкоманд и их параметров (read и write API)

## Действия
1. Описать перечень подкоманд по `docs/DocsWalker.yml` (один источник правды).
2. Реализовать парсер argv: первая позиционная — имя команды, далее именованные `--key=value`, списки через запятую. Имена параметров — kebab-case (например, `--parent-id`, `--from-id`).
3. Общий параметр `--root=<path>`: путь к корню проекта (где лежит `docs/`). По умолчанию — поиск ближайшего `docs/` вверх по дереву от текущей директории; если не найден — структурированная ошибка с кодом `docs_not_found`.
4. Общая форма ответов: успех в stdout — JSON-объект с полем `result` (или просто результат), ошибка в stderr — `{ "ok": false, "error": { "code", "path", "message" } }`, exit code 0 для успеха, ненулевой для ошибки.
5. Каждая команда из таблицы возвращает `not_implemented` в форме структурированной ошибки.
6. Сериализация — `System.Text.Json` с source-генераторами (`JsonSerializerContext` с явно перечисленными DTO ответов и ошибок) — обязательное условие AOT-совместимости. snake_case-имена полей через `JsonNamingPolicy`. Для динамических `result`-полей произвольной формы — `JsonNode`/`JsonElement`, не `object` через рефлексию.

## Риски
- Параметры `transaction` сложные (массив операций) — на этом шаге достаточно описать их в таблице как «принимается JSON-строка», детальный разбор — в `write-api-transaction`.
- Поведение `--root` отразить в `docs/DocsWalker.yml`/«CLI-интерфейс» синхронно с реализацией.
