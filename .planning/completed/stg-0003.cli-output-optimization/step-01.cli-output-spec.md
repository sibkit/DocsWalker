# stg-0003 — cli-output-spec

## Цель
Зафиксировать новый контракт CLI-ответа в `docs/` (единственный источник правды по проекту). Шаг не трогает C#-код. Все последующие шаги (`drop-cli-envelope`, `get-refs-map-form`, `omit-defaults-empties`, `trim-error-describe-type`) опираются на эту правку как на спецификацию.

## Файлы

- `docs/DocsWalker.yml` — основная нагрузка шага. Узлы про CLI-интерфейс (секция `CLI-интерфейс`, корень id=35) и узлы LLM-guide (секция `Как LLM работает с DocsWalker`, корень id=274). Точные id перепроверить через `docswalker get-map` перед правкой — id могли смещаться.
- `docs/Схема.yml` — править только если в типах сейчас явно описан старый формат ответов. По модели проекта Схема описывает узлы и связи, не CLI-контракт; скорее всего трогать не придётся. Решение по факту — после чтения.
- `docs/.docswalker/meta-schema.yml` — **не править на этом шаге**. Вопрос про `default: many` / `default: false` для `cardinality`/`required` — относится к шагу `omit-defaults-empties` и решается там. Здесь меняем только спецификацию формата ответов.

## Действия

1. Через CLI (`docswalker get-subtree --id=35` и `docswalker get-subtree --id=274` или прямым чтением YAML) ознакомиться с текущим описанием CLI-интерфейса и LLM-guide. Понять, какие конкретно атомарные узлы (rule/statement/example/note/llm_hint) описывают:
   - формат ответа CLI (старый: `{ok, result}` / `{ok, error: {…}}`),
   - форму `get-refs` (старая: `in[]`/`out[]` массивы записей `{name, target_id}`),
   - форму embedded `describe_type` в ошибках (старая: полный контракт типа),
   - правила сериализации (если описано — про обязательное присутствие `cardinality`/`required`/`children`/`subtree_tokens`/`parameters`/`examples`).

2. Переписать соответствующие узлы под новый контракт. Использовать write-команды CLI (`update-node` для правки text, `create-node`/`delete-nodes` если нужны новые/убираются старые узлы). Прямую правку YAML не делать — нарушение правила проекта.

   **Что должно быть зафиксировано в LLM-guide и/или CLI-интерфейс-секции `docs/DocsWalker.yml`:**

   **(a) Конверт CLI-ответа.**
   - Успех: exit 0, stdout — JSON-объект (или массив на верхнем уровне для команд, чей результат — список), являющийся непосредственно результатом команды. Никакой обёртки.
   - Для write-команд в результате присутствует поле `applied: bool` верхнего уровня. true — изменения записаны на FS; false — это был dry-run.
   - Ошибка: exit ≠ 0, stderr — JSON-объект с полями `code`, `message`, опционально `path`, `hint`, `describe_type`. Никакого вложенного объекта `error`.
   - stdout при ошибке пустой; stderr при успехе пустой.
   - Дискриминатор успех/ошибка — exit-code (поток stdout/stderr дублирует тот же сигнал).

   **(b) Форма `get-refs`.**
   - Результат — объект с двумя полями `in` и `out`. Каждое — map имени связи в массив id целей: `{"<ref-name>": [target-ids…]}`.
   - Та же форма, что у `out_refs` в `get-nodes` и `describe-type` — единый контракт по всей API.
   - Параметр `--name=<filter>` оставляет в обеих map'ах только запись с этим именем; если связей с таким именем нет — обе map'ы пустые объекты.
   - Порядок id внутри списка — по возрастанию.

   **(c) Embedded `describe_type` в ошибках записи.**
   - При ошибках с локализацией на конкретной связи (`missing_required_ref`, `invalid_ref_value` и аналоги) поле `error.describe_type.out_refs` содержит только проблемную связь — одну запись.
   - Шапка типа (`name`, `description`, `text_required`) присутствует всегда.
   - LLM, нуждающаяся в полном контракте типа, обязана вызвать `describe-type --name=<X>` явно.

   **(d) Опускаемые дефолты и пустоты.**
   - `cardinality:"many"` и `required:false` в RefDef — не сериализуются (дефолты). Отсутствие поля при чтении трактуется как дефолт.
   - `children: []` в результате `get-map`/`get-subtree` — не сериализуется. Отсутствие поля `children` ⇒ узел-лист.
   - `subtree_tokens`, равный `tokens` — не сериализуется. Отсутствие поля `subtree_tokens` ⇒ узел листовой в текущем поддереве, и `subtree_tokens = tokens`.
   - `parameters: []` и `examples: []` в `get-usage-guide` — не сериализуются. Отсутствие поля ⇒ команда без параметров / без примеров.

3. Удалить старые формулировки про `{ok, result}` / `{ok, error: {…}}` / плоский `in/out`-массив / полный embedded describe_type — не оставлять «historical» комментарии. Расхождение docs ↔ код в любом виде запрещено правилом проекта.

4. После всех правок прогнать `docswalker check-integrity` — должна вернуть пустой `errors`. Если интегральные правила перестают проходить (например, удалённые узлы оставили dangling refs) — диагностировать и доправить связи через `redirect-refs` / `delete-ref`.

5. Закоммитить пачкой по конвенции проекта: один commit message на английском в imperative, push в `origin/master`. Атомарные git-вызовы (правило проектного `CLAUDE.md`).

## Риски

- Встроенные узлы `examples` под правилами могут потребовать обновления — если в живых docs есть пример старого формата CLI-ответа в text узла example. Обновить такие примеры под новый формат.
- Узлы LLM-guide, описывающие порядок «как LLM работает с CLI», могут противоречить новым формулировкам. Не оставлять оба варианта — выбирать одну версию (новую) и удалять старую.
- Если по факту чтения окажется, что `docs/Схема.yml` тоже описывает CLI-контракт (через description у типов или RefDef) — править и его. Если нет — не трогать.
