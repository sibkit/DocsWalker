# stg-0003 — get-refs-map-form

## Цель
Перевести `get-refs` с формы плоских массивов `in[]`/`out[]` записей `{name, target_id}` на форму map'ов `{in: {<ref-name>: [target-ids…]}, out: {<ref-name>: [target-ids…]}}`. Совпадает с формой `out_refs` в `get-nodes` и `describe-type` — единый контракт «связь → список целей» по всей API.

## Файлы

- `src/DocsWalker.Core/Api/ReadApi.cs` — метод `GetRefs` или соответствующий DTO. Точное место уточнить через Glider: `mcp__glider__search_symbols GetRefs kinds=Method`. Прочитать `mcp__glider__get_method_source`.
- `src/DocsWalker.Core/Api/ReadApiJson.cs` (или другой JSON-сериализующий слой Core) — там, где `in[]`/`out[]` собираются как `JsonArray` записей. Перевести на `JsonObject` с группировкой по имени связи.
- `src/DocsWalker.Cli/Cli/Handlers/ReadHandlers.cs` — handler `GetRefs`. Если форма формируется в Core — handler не правится; если форма формируется в Cli-слое — править здесь. Уточнить чтением.
- Тесты — `tests/DocsWalker.Core.Tests/` или `tests/DocsWalker.Cli.Tests/`. Поиск через Grep по подстроке `target_id` или `\"name\":\"sections\"` в файлах тестов.

## Действия

1. Через Glider найти `GetRefs` в Core: `mcp__glider__find_code GetRefs intent=symbol`. Прочитать сам метод и его JSON-сериализатор.

2. Понять текущую форму. Со слов выводов CLI: результат — объект, поля `in` и `out` — массивы объектов `{name, target_id}` без какой-либо группировки. На узле id=1 в `out` 16 записей, имя `sections` повторяется 15 раз.

3. Перевести сборку результата на map-форму:
   - для каждого направления (in / out) пройти по записям, сгруппировать по имени связи;
   - для каждого имени собрать массив id целей, отсортировать по возрастанию;
   - результат — `JsonObject` `{<ref-name>: JsonArray из id, …}`.
   - финальный результат: `{in: <map>, out: <map>}`.

4. Параметр `--name=<filter>`:
   - если задан — после группировки в обеих map'ах оставить только запись с этим именем;
   - если связей с таким именем нет ни в одном направлении — обе map'ы пустые объекты `{}` (не отсутствующие поля). Пустой `{}` — валидный сигнал «направление есть, ref'ов нужного имени нет».

5. Тесты: переписать эталоны под новый формат. Проверить сценарии:
   - `get-refs --id=<root>` — на корне граф (root id=0) только path-связи входящие, проверить структуру;
   - `get-refs --id=<документ>` — много входящих/исходящих с группировкой;
   - `get-refs --id=<id> --name=<имя>` — фильтр оставляет одну запись в каждой map'е;
   - `get-refs --id=<id> --name=<несуществующее_имя>` — обе map'ы пустые `{}`.

6. Прогнать вручную:
   - `docswalker get-refs --id=1` — должна вернуть compact map-form без дублирования имён;
   - `docswalker get-refs --id=1 --name=sections` — `out` содержит только запись `sections`, `in` пустой `{}`;
   - `docswalker get-refs --id=42 --name=examples` — на правиле, у которого есть `examples` cross-ref.

7. Если документация в `docs/DocsWalker.yml` уже обновлена шагом `cli-output-spec` — пропустить; если нет (по факту шага 1) — синхронизировать здесь же.

8. Закоммитить + push.

## Риски

- **Внешние консьюмеры старой формы.** Проект ранний, внешних консьюмеров нет. Это разовый breaking change, фиксируется в `docs/` (см. `cli-output-spec`).
- **Стабильный порядок ключей в JsonObject.** `System.Text.Json.Nodes.JsonObject` сохраняет порядок вставки. Чтобы вывод был детерминированным — вставлять имена связей в map в стабильном порядке (например, по алфавиту). Это не критично для корректности, но важно для предсказуемых diff'ов в тестах.
- **`ReadApi.GetRefs` может возвращать DTO, сериализуемый source-generated context'ом.** Если так — придётся либо менять DTO (поля типа `Dictionary<string, List<int>>`), либо перевести сериализацию на ручной `JsonObject`. Выбор — по факту чтения; обе ветки рабочие.
