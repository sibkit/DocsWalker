# stg-0011 — code-raise-text-limit

## Цель

Поднять предельную длину `text` узла с 1000 до 2000 символов. По ходу шагов 01–05 несколько расширенных API-definition'ов (`search`, `get_tree`, `get_overview`) упёрлись в лимит и пришлось сокращать формулировки; новый предел 2000 — компромисс между плотностью узла и риском мегапростыней.

## Файлы

- `src/DocsWalker.Kernel/**` — место, где живёт валидатор `text_too_long`; точная локация определяется через Glider `search_text "text_too_long"`.
- Сопутствующие тесты в `tests/**` (если есть проверки лимита) — поднять граничные значения.
- Publish-папки `src/DocsWalker.Kernel/bin/.../publish/` и `src/DocsWalker.Cli/bin/.../publish/` — пересобрать exe после правки.

## Действия

1. Glider `search_text "text_too_long"` → найти узел с константой лимита.
2. Заменить 1000 → 2000.
3. Glider `find_code` / `search_text` по «1000» в окрестных файлах валидации — убедиться, что больше нигде это число не зашито.
4. Обновить тесты (если есть): пограничные значения, сообщения об ошибке.
5. `dotnet publish` для `DocsWalker.Kernel` и `DocsWalker.Cli` под `win-x64 Release`.
6. Остановить старый kernel (`taskkill /F /PID …`), запустить новый.
7. Smoke: через CLI `update-node` записать в любой узел text длиной 1500 символов — должен пройти; 2500 — должен упасть с обновлённым лимитом в сообщении.

## Риски

- Тест может быть параметризован константой из самого кода — тогда менять её не надо.
- Старая запись в `docs/` с text < 1000 не пострадает; узлы остаются валидными.
- Если AOT-publish упирается в .NET 10 SDK — нужно его наличие; CLAUDE.md фиксирует требование.
